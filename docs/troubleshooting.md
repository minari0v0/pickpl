# PickPl Troubleshooting Log

이 문서는 PickPl 프로젝트 개발 중 발생한 주요 오류 현상과 원인 분석, 그리고 해결 과정을 기록한 트러블 슈팅 로그입니다.

---

## 프론트엔드 (Frontend)

### 관리자 인증 비밀번호 페이지 로그인 무한 루프
* **오류 현상**: 관리자 비밀번호를 정상적으로 입력하고 로그인에 성공했음에도 불구하고, 대시보드로 넘어간 후 다시 비밀번호 폼 페이지로 되돌아가는 현상이 무한 반복됨.
* **원인**: 로그인 성공 시 입력받은 시크릿 키가 세션스토리지(`sessionStorage`)에 정상 저장되지 않았거나, 이후 대시보드에서 백엔드 데이터를 요청할 때 HTTP 헤더(`X-Admin-Secret-Key`)에 시크릿 키가 누락되어 403 Forbidden 에러가 발생, 프론트엔드가 권한 없음으로 판단하여 로그인 페이지로 리다이렉트시켰기 때문임.
* **해결 방법**: 로그인 폼 서브밋 시 `sessionStorage.setItem('adminSecretKey', passwordInput.trim())`를 통해 브라우저에 인증 키를 확실히 안전하게 저장하고, 대시보드 및 각 탭 컴포넌트에서 API 호출 시 헤더에 해당 값을 항상 실어 보내도록 코드를 보강함.

### PickPl 로고 줄바꿈 및 자간 어긋남 현상
* **오류 현상**: 사이드바 LNB 로고나 로그인 폼의 브랜딩 로고 `PickPl`이 `Pick Pl`처럼 띄어쓰기가 들어간 형태로 나타나 자간 및 정렬이 깨져 보임.
* **원인**: React JSX 내에서 `<h1 className="font-logo">Pick<span>Pl</span></h1>` 등의 코드를 개행(줄바꿈)해서 작성하는 경우, 리액트 컴파일러와 브라우저 엔진이 태그 사이의 공백(White space)을 문자 그대로의 띄어쓰기로 취급하여 렌더링하기 때문임.
* **해결 방법**: `Pick<span>Pl</span>` 형태로 코드 내부의 줄바꿈과 공백을 완전히 붙여 써서 브라우저가 추가 공백을 만들지 않도록 교정함.

### 신규 데이터 적재 드래그 앤 드롭 미작동 및 파일 열기 이동 현상
* **오류 현상**: 크롤러 분석 JSON 파일을 브라우저 화면의 드롭 영역에 놓았을 때, 적재 탭에 파일이 로드되지 않고 브라우저 탭 자체가 JSON 텍스트 뷰어 화면으로 강제 이동해 버림.
* **원인**: 드래그 앤 드롭 이벤트를 바인딩할 때 브라우저의 기본 파일 수신 및 페이지 이동 동작(`Default Action`)을 막지 않아 브라우저가 파일을 직접 열어버렸기 때문임.
* **해결 방법**: 드롭존에 연결한 `onDragOver`, `onDragLeave`, `onDrop` 이벤트 핸들러의 초입에 `e.preventDefault()`와 `e.stopPropagation()`을 필수 적용하여 기본 브라우저 동작을 차단하고, 드롭된 `e.dataTransfer.files` 목록을 끄집어내 `FileReader`를 통해 메모리로 직접 불러오도록 수정함.

## 백엔드 (Backend)

### 일괄 공개 전환 및 다중 공개 설정 API 미매핑 (HTTP 405 Method Not Allowed)
* **오류 현상**: 관리자 설정 탭에서 비공개 장소들을 '일괄 공개 전환' 하거나, 공간 관리 탭에서 다수의 장소를 선택해 일괄 활성화를 시도할 때 화면에 `⚠️ 일괄 공개 처리 실패: HTTP 405` 또는 HTTP 405 Method Not Allowed 에러가 노출됨.
* **원인**: 프론트엔드는 `POST /api/v1/admin/places/publish-all` 및 `POST /api/v1/admin/places/bulk-publish-status` 엔드포인트를 호출하려 했으나, 백엔드 컨트롤러에 해당 URL을 매핑하는 컨트롤러 핸들러 및 DTO 구조가 구축되어 있지 않아 백엔드가 잘못된 HTTP 메서드 요청으로 간주했기 때문임.
* **해결 방법**: `AdminPlaceController.java`에 일괄 공개를 담당하는 `publishAllPlaces` POST 핸들러와 대량 변경을 처리하는 `updatePublishStatusBulk` POST 핸들러 및 바인딩용 DTO(`BulkPublishRequest`)를 구성하고, `PlaceService.java`에 JPA 기반 벌크 벌 업데이트 및 목록 갱신 비즈니스 로직을 설계하여 연동함.

### 단건 공간 정보 수정 시 HTTP 500 Internal Server Error (유니크 제약조건 위반)
* **오류 현상**: 관리자 공간 관리 탭에서 장소를 개별 클릭해 '에디터의 한 마디'나 태그 등을 수정하고 적용하려 할 때, 혹은 '비공개' 배지를 클릭해 공개 상태를 토글하려 할 때 `수정 실패/상태 변경 실패: HTTP 500` 오류가 화면에 발생함.
* **원인**: 태그 관계를 갱신할 때 기존의 중간 매핑 엔티티 컬렉션을 완전히 비우는 `place.getPlaceTagMaps().clear()`를 실행하더라도, JPA/Hibernate의 내부 트랜잭션 쓰기 지연(ActionQueue) 동작 순서상 기존 매핑의 `DELETE` 쿼리보다 컬렉션에 임시로 추가된 새 태그들의 `INSERT` 쿼리가 데이터베이스에 먼저 발송됨. 이로 인해 동일한 `(place_id, tag_id)` 조합에 대해 복합 유니크 제약조건 `uk_place_tag` 충돌이 먼저 유발되는 한계 때문임.
* **해결 방법**: 무조건적인 컬렉션 초기화 방식 대신, 기존에 할당된 태그 목록과 새로 들어온 태그 목록을 비교(Diff/Compare)하는 차분(Delta) 업데이트 방식을 적용함. 삭제할 대상만 `removeTag()`로 솎아내고, 신규 추가할 대상만 `addTag()`로 주입함으로써 데이터베이스 쿼리를 최소화하고, 변경되지 않는 기존 태그 매핑은 그대로 보존해 유니크 키 충돌을 100% 예방함.

### 관리자 공간 일괄 삭제 시 HTTP 500 (외래키 제약조건 위반 에러)
* **오류 현상**: CMS 공간 관리 탭에서 다수의 공간을 체크박스로 선택하고 일괄 삭제할 때 `일괄 삭제 실패: HTTP 500`과 함께 `SQLIntegrityConstraintViolationException` 및 외래키 오류(Cannot delete or update a parent row: a foreign key constraint fails)가 발생함.
* **원인**:
  1. `deleteAllByIdInBatch(ids)`와 같은 벌크 쿼리 메서드는 Hibernate의 영속성 컨텍스트 생명주기(JPA lifecycle)를 완전히 우회하여 direct SQL `DELETE`를 실행하므로, `Place` 엔티티 내에 설정된 `@OneToMany(cascade = CascadeType.ALL)` 설정이 동작하지 않아 연관 매핑 데이터(`PlaceTagMap`, `Scrap`)가 그대로 남은 채 부모를 지우려 했기 때문임.
  2. 또한 `VibeVote` 엔티티는 `Place`를 단방향으로 참조하고 있어 `Place` 측에서 Cascade 설정을 걸어둘 수 없으므로, 해당 장소에 투표 내역이 하나라도 존재하는 경우 데이터베이스 제약 조건상 삭제가 무조건 차단되는 구조적 원인 때문임.
* **해결 방법**: 
  1. 벌크 삭제 실행 시, 먼저 관련 `VibeVote` 데이터를 `vibeVoteRepository.deleteByPlaceIdIn(ids)`를 통해 명시적으로 사전 삭제하도록 구성함.
  2. 벌크 쿼리(`deleteAllByIdInBatch`) 대신, 영속 객체 목록을 로드하여 `placeRepository.deleteAll(places)`를 실행함으로써 JPA Cascade 영속성 전이를 유도하여 연관된 `PlaceTagMap`, `Scrap` 데이터가 먼저 깨끗하게 청소된 후 정상 삭제되도록 조치함.

### 페이징 무한 스크롤 도입 시 JPQL 비표준 LIKE 구문으로 인한 백엔드 컨텍스트 초기화 실패 및 프론트 데이터 증발 현상
* **오류 현상**: 페이징 및 무한 스크롤 기능을 도입한 후 메인 화면(홈/발견)에 처음 접속하거나 F5 새로고침을 하면 약 0.1초 동안만 데이터가 잠깐 노출되었다가 즉시 사라진 뒤, `"데이터가 없습니다. 백엔드를 확인해주세요."` 문구가 표시되고 더 이상 장소 리스트가 노출되지 않으며 스크롤 로딩도 되지 않는 현상. 백엔드 터미널에서도 2~3회 조회 로그가 찍힌 후 먹통이 됨.
* **원인**:
  1. **백엔드 원인**: `PlaceRepository.java` 내에 추가된 페이징용 JPQL 쿼리(`findPlacesByKeyword`, `findPlacesMatchingAllTagsAndKeyword`) 작성 시 `LIKE %:keyword%`와 같은 비표준 바인딩 형식을 사용함. 이로 인해 Spring Boot 기동 시 Hibernate의 JPQL 파서가 구문 검증(Query Syntax Validation) 과정에서 에러를 유발해 애플리케이션 컨텍스트 초기화가 정상적으로 완수되지 않고 예외가 발생함.
  2. **프론트엔드 원인**: 브라우저에서 최초 접속 시 SSR을 통해 초기 데이터를 0.1초 동안 표시하는 것에는 성공하지만, 컴포넌트 마운트가 완료되자마자 브라우저의 GPS Geolocation API 센서 작동(또는 Fallback 좌표 로딩)에 의해 사용자 위경도 정보가 업데이트됨. 이에 따라 필터 기준이 변경되어 `isInitialLoad` 조건이 `false`로 풀리게 되고 `page=0`에 대한 SWR API 갱신 요청이 백엔드로 즉시 날아감. 그러나 백엔드 서버가 위 JPQL 구문 오류로 인해 비정상 상태이거나 쿼리 실행 실패를 반환하므로, SWR이 빈 응답 또는 에러를 반환받아 화면상에서 기존에 표시되던 SSR 데이터(`initialPlaces`)를 싹 덮어써 증발시키는 현상이 일어남.
* **해결 방법**:
  1. `PlaceRepository.java`의 JPQL 문법 내 `LIKE %:keyword%`로 선언된 모든 구문을 데이터베이스 및 Hibernate 표준 형식인 `LIKE CONCAT('%', :keyword, '%')` 형태로 치환함.
  2. Gradle 빌드 명령(`.\gradlew.bat compileJava`)을 실행해 수정된 쿼리 선언 및 컴파일에 오류가 없음을 1차 확인한 뒤, 사용자에게 작동 중인 백엔드 서버 프로세스(`make back`)를 완전히 끄고 재기동하도록 가이드하여 수정을 백엔드 런타임에 최종 반영함. 이후 F5 새로고침 시 무한 스크롤 및 초기 로딩이 매끄럽고 안정적으로 작동하는 것을 검증함.

## 데이터 크롤링 (Data Crawling / Pipeline)

### 외부 플레이스 상세 정보 수집 차단 (403 Forbidden / CAPTCHA 이슈)
* **오류 현상**: 로컬 환경에서 네이버/카카오 플레이스 상세 스크래핑 파이프라인을 구동할 때, 상세 평점이나 이미지 데이터가 빈 값으로 수집되거나 지속해서 HTTP 403 차단 오류가 반환됨.
* **원인**: 외부 포털 플랫폼에서 스크래핑 도구(Playwright 및 HTTP 클라이언트 라이브러리)의 User-Agent 정보나 요청 헤더를 봇(Bot)으로 감지하여 차단 로직(CAPTCHA 또는 IP 임시 차단)을 발동시켰기 때문임.
* **해결 방법**: Playwright 및 크롤러의 브라우저 컨텍스트 설정에 일반 데스크톱 및 모바일 크롬 브라우저의 실제 헤더와 동일하게 User-Agent를 조작(Spoofing)하고, 요청 시 무작위의 시간 지연(Random Delay)과 모바일 뷰 전용 헤더 속성들을 결합해 타겟 서버의 탐지 차단 정책을 안정적으로 우회함.

### Gemini API 429 무료 쿼터 초과 시 파이프라인 무한 로딩 및 데이터 유실
* **오류 현상**: 데이터 파이프라인(크롤러 및 AI 분석) 구동 중 Gemini API의 무료 일일 한도(429 `RESOURCE_EXHAUSTED`)를 초과했을 때, 분석 작업이 무한 로딩/대기 상태에 빠지거나 여태까지 가공된 데이터가 디스크에 저장되지 못하고 소실되는 버그가 일어남.
* **원인**: 이전 수집 엔진은 크롤링과 AI 분석이 단일 실행 스레드로 강하게 결합되어 있었으며, 예외 처리가 부실하여 429 한계 에러 발생 시 진행도가 실시간 디스크로 커밋되지 못해 누락이 발생함.
* **해결 방법**: 
  1. 수집(1단계 `--scrape`)과 AI 분석(2단계 `--analyze`) 단계를 물리적으로 분리하여 느슨한 결합으로 전환함.
  2. Gemini AI 호출 부에 429 감지 시 25초 대기 후 최대 3회 점진적 백오프(Progressive Backoff) 재시도 로직을 설계하고, 최종 불가 시 `QuotaExhaustedError` 예외를 명시적으로 상향 전파함.
  3. AI 분석 루프에서 해당 예외 포착 시, **그 시점까지 성공 완료된 배치 분석 데이터들만 실시간으로 즉시 병합(`save_and_merge_results`)하여 안전하게 디스크에 기록**하고 프로세스를 조기 정상 종료(`sys.exit(2)`) 처리함.
  4. 재실행 시 기존 JSON에 누적 완료된 장소 ID들을 우선 감지해 자동으로 필터링 및 건너뛰는 **이어서 수집/분석(Resume) 기술**을 완성하여 데이터 무손실과 연장선 상의 연속성을 보장함.

### Gemini API 대기 중 Tkinter GUI 모니터 얼어붙음 (Not Responding)
* **오류 현상**: 데이터 수집, AI 분석, 백엔드 적재 등의 파이프라인 구동 시, 팝업된 GUI 모니터 창이 "응답 없음" 상태로 얼어붙거나 화면이 하얗게 굳어서 실시간 상태 및 로그 스트리밍을 파악할 수 없는 현상 발생.
* **원인**: GUI를 그리고 윈도우 OS의 창 관리 이벤트에 대응해야 하는 메인 스레드가 이미지 다운로드 및 Gemini API의 동기식 HTTP 네트워크 대기(10~30초)를 직접 수행하면서 이벤트 루프(`root.mainloop()`)가 블락(Block)되었기 때문임.
* **해결 방법**: 연산 프로세스와 UI 프로세스를 완전히 분리함. 메인 스레드는 오직 GUI 이벤트 루프만 전담 구동하게 하고, 무거운 데이터 연산(수집, 분석 등)은 백그라운드 데몬 스레드(`Daemon Thread`)로 띄워 실행시킴. 또한 백그라운드 스레드에서 UI 요소를 안전하게 제어하도록 Tkinter의 `root.after()` 메시지 큐를 통한 Thread-Safe 위젯 업데이트 방식으로 전면 개편함.

### Gemini API 503 서버 오류 중단 및 429 초과 후 무의미한 재요청 누적 지연
* **오류 현상**: Gemini API 무료 티어 사용 중 서버 부하 폭주로 인한 `503 Service Unavailable` 및 `UNAVAILABLE` 에러 발생 시 분석이 즉시 중단되거나, 한 번 `429 RESOURCE_EXHAUSTED`가 발생한 상태에서도 다음 배치마다 불필요하게 3-flash-preview 호출을 재시도하면서 매번 25~50초의 극심한 대기 지연(Delay)이 매 루프마다 축적되는 현상 발생.
* **원인**: 503 서버 오류가 발생했을 때 자동 재시도 예외 처리가 부재했고, 429 한도 초과는 자정 리셋 전까지 복구 불가능함에도 매 배치마다 3-flash-preview 모델을 1차 시도로 찔러본 뒤 2.5-flash로 폴백하도록 설계되었기 때문임.
* **해결 방법**: 
  1. 503 및 UNAVAILABLE 오류 발생 시에도 즉시 죽지 않고 429처럼 25초, 50초 단위의 점진적 백오프 후 최대 3회 자동 재시도하는 예외 복구 로직을 구현함.
  2. 한 번이라도 429 쿼터 초과가 감지되면 해당 실행 생명주기 동안 `use_fallback_model` 스티키 플래그를 `True`로 고정함. 이후 배치 분석에서는 한도가 끝난 3-preview 모델 호출을 생략하고 처음부터 `gemini-2.5-flash` 모델을 즉시 다이렉트 호출하도록(Sticky Fallback) 개선하여 낭비되는 지연 시간을 원천 차단함.

### Playwright 크롤러를 통한 Naver Place 위경도 추출 실패 및 디폴트 좌표 고정
* **오류 현상**: 파이프라인 좌표 수집 동작 시 일부 장소(예: 헬로오드리, 서울로인 등)의 좌표가 유효한 값 대신 디폴트 좌표인 `37.55, 126.92`로 일괄 강제 설정되는 현상이 발생함.
* **원인**: 
  1. 수집기에서 성능 고속화를 위해 Playwright `page.goto` 호출 시 `"commit"` 옵션을 주어 HTML 조각을 파싱하도록 최적화하였으나, 특정 장소의 경우 SSR(Server-Side Rendered) HTML 본문에 Apollo State 좌표 데이터가 아예 누락된 상태로 넘어오는 경우가 있음.
  2. 해당 공간들은 JavaScript 로딩 완료 후에 동적으로 지도 영역과 길찾기 딥링크(`dlat`/`dlng`)가 렌더링되는데, 웹드라이버가 DOM 완성 전에 즉시 좌표를 추출하려다 매칭에 실패하여 Fallback 기본 위경도가 강제 주입되었기 때문임.
* **해결 방법**:
  1. 일차적으로 `"commit"` 시점에 SSR HTML을 긁어 빠르게 파싱하되, 매칭 실패 시 이를 즉각 감지하여 2단계 폴백(Fallback) 프로세스를 구동하도록 보완함.
  2. 폴백 프로세스에서는 Playwright의 `wait_for_load_state("domcontentloaded")`를 호출하여 DOM 트리를 완전히 수신하고 추가로 2초의 대기 지연 시간을 확보하여 동적 내비게이션 요소가 온전히 렌더링되게 함.
  3. 그 후 HTML 소스 내에서 `dlat=([\d\.]+)` 및 `dlng=([\d\.]+)` 정규식을 활용해 정확한 위도/경도를 파싱해내도록 예외 처리 체계를 고도화하여 수집률 100%를 달성함.
